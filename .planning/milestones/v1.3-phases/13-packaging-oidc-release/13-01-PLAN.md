<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| release workflow → PyPI/TestPyPI | CI identity crosses into the public package index; a mis-scoped publish job or leaked token would let an attacker publish under the project's name |
| repo docs/metadata → installer | pyproject/README/CHANGELOG drive what users `pip install`; drift can misdirect installs to a wrong or squatted name |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Spoofing | release.yml publish jobs | mitigate | Task 2 asserts id-token: write present on every publish job so OIDC Trusted Publisher identity is enforced (no ambient token) |
| T-13-02 | Tampering | release.yml credential regression | mitigate | Task 2 fails if any stored PyPI/TestPyPI token secret or `password:` input reappears — blocks silent revert to long-lived token auth |
| T-13-03 | Spoofing | distribution identity drift | mitigate | Task 1 pins pyproject name == kafka-events-mcp and cross-checks README/CHANGELOG so a rename/typosquat mismatch fails CI |
| T-13-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan — tests use stdlib (tomllib) + already-present pyyaml; build smoke check is opt-in/skippable, so no install path is added |
</threat_model>
