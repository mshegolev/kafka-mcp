# Phase 8 Plan: Multi-Topic Search & Header Filtering

## Goal
Enable investigators to search for a key across multiple Kafka topics in a single call and filter results by header key-value pairs, getting a merged-by-timestamp result set — without breaking any existing single-topic callers.

## Success Criteria
1. `KafkaClient.search_messages(key="order-123", topics=["orders", "payments", "shipments"])` returns messages from all three topics merged and sorted by `timestamp_utc`; each result carries its source `topic` field — verifiable via `pytest tests/ -k "multi_topic"` with mocked consumer returning messages from different topics
2. `KafkaClient.search_messages(key="order-123", topics=["orders"])` produces identical results to the pre-v1.2 single-topic path; existing tests in `tests/` that call `search_messages` with a single topic pass without modification — verifiable via `pytest tests/ -k "search"` showing zero regressions
3. `KafkaClient.search_messages(key="order-123", headers={"trace_id": "abc-123"})` returns only messages whose Kafka headers contain the specified key-value pairs; messages missing the header or having a different value are excluded — verifiable via `pytest tests/ -k "header_filter"`
4. Header filtering combines with key + time window + multi-topic filters using AND semantics: `search_messages(key="order-123", topics=["orders", "payments"], headers={"trace_id": "abc"}, time_from=..., time_to=...)` applies all filters simultaneously — verifiable via `pytest tests/ -k "combined_filter"`
5. All new parameters (`topics`, `headers`) are optional with backward-compatible defaults (`topics=None` falls back to the existing single-topic `topic` parameter; `headers=None` means no header filtering) — verifiable via `pytest tests/ -k "search" --tb=short` showing full green on existing + new tests

## Implementation Tasks

### 1. Domain Layer Changes
- **Add `_matches_headers` helper function** in `src/kafka_mcp/domain/search_service.py`:
  - Takes a `KafkaMessage` and a `dict[str, str]` of header key-value pairs
  - Returns `True` if all specified headers match exactly, `False` otherwise
  - Handles `None` and empty dict cases (returns `True` for no filtering)

- **Modify `search_messages` method** in `src/kafka_mcp/domain/search_service.py`:
  - Add `headers: dict[str, str] | None = None` parameter
  - Update docstring to document the new parameter
  - Apply header filtering in the message matching logic using AND semantics
  - Sort final results by `timestamp_utc` across all topics

### 2. Library Adapter Changes
- **Update `search_messages` method** in `src/kafka_mcp/adapters/inbound/lib.py`:
  - Update docstring to include `headers` in the supported keyword args list

### 3. CLI Changes
- **Add `--headers` argument** to the `search-messages` subcommand in `src/kafka_mcp/adapters/inbound/cli.py`:
  - Accepts comma-separated key-value pairs (e.g., `trace_id=abc,key2=value2`)
  - Add help text explaining the format

- **Update `run_search_messages` function** in `src/kafka_mcp/adapters/inbound/cli.py`:
  - Add `headers: str | None = None` parameter
  - Parse the headers string into a dictionary
  - Pass the parsed headers dict to the client's `search_messages` method

- **Update `main` function** in `src/kafka_mcp/adapters/inbound/cli.py`:
  - Pass the `headers` argument from the parsed namespace to `run_search_messages`

### 4. Testing
- **Add unit tests for `_matches_headers`** in `tests/test_domain.py`:
  - Test `None` and empty dict cases
  - Test single header matching
  - Test multiple header matching
  - Test mismatch cases

- **Add integration tests for header filtering** in `tests/test_domain.py`:
  - Test header filtering with single topic
  - Test header filtering with multiple topics
  - Test combined key + header filtering
  - Test multi-topic results sorting by timestamp

- **Add CLI tests** in `tests/test_inbound.py`:
  - Test CLI argument parsing for `--headers`
  - Test CLI headers string parsing logic

## Backward Compatibility
- All existing calls to `search_messages` continue to work unchanged
- The `topics` parameter was already implemented in previous versions
- The `headers` parameter defaults to `None`, maintaining existing behavior
- Existing tests continue to pass

## Performance Considerations
- Header filtering is applied after key matching for efficiency
- Multi-topic results are sorted by timestamp after collection
- No parallel processing is implemented in this phase (reserved for future optimization)
