<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| raw_key bytes → _extract_schema_id | Attacker-controlled wire bytes; integer extraction must be length-guarded |
| raw_key bytes → _decode_key → SchemaRegistryPort | Attacker-controlled framing passed to SR decode; DecodeError must be swallowed |
| inbound faces serialization | raw_key bytes base64-encoded before leaving the domain boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering | _extract_schema_id(raw_key) | mitigate | Length guard `len(raw) >= 5` checked BEFORE `raw[0]` read and BEFORE `raw[1:5]` slice. Short/malformed key bytes return None without panic. |
| T-04-04 | Denial of Service | _decode_key — unbounded SR decode time | mitigate | Reuses existing SchemaRegistryHttpAdapter.decode() which already has typed DecodeError handling. DecodeError is caught and swallowed — the message is never dropped. No timeout is added here (SR timeout is an existing concern, not introduced in Phase 4). |
| T-04-05 | Information Disclosure | key_decoded dict flows through all four faces | accept | key_decoded is surfaced only when it was already decodable from the SR (same posture as value decode). No new data is surfaced beyond what the SR already returns for value decode. |
| T-04-06 | Tampering | raw_key base64 in serializers | accept | base64 is a reversible encoding — caller receives the raw bytes representation. This is the same pattern as the existing `raw` field. No risk beyond the existing raw field posture. |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan. Existing dependencies unchanged. |
</threat_model>
