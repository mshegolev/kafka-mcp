## Fixed Issues

### WR-01: Generic Protobuf decode introduced an undeclared runtime dependency on a `protoc` system binary

**Files modified:** `src/kafka_mcp/adapters/outbound/schema_registry_http.py`, `pyproject.toml`, `tests/test_adapters.py`
**Commit:** 0065786
**Applied fix:** Replaced the `shutil.which("protoc")` + `subprocess.run([...])` system-binary
invocation with the in-process, wheel-bundled compiler from `grpcio-tools`
(`grpc_tools.protoc.main([...])`). No external `protoc` on PATH is required, so the brick stays
`pip install`-able into slim containers / bare venvs. Added `grpcio-tools>=1.60` to
`[project].dependencies`. The test helper `_build_protobuf_payload` was switched to the same
in-process compiler, and the class-level `skipif` was changed from "system protoc present" to
"grpcio-tools importable" so the decode path is now exercised without a PATH binary. Removed the
now-unused `shutil`/`subprocess` imports from the adapter and the unused `shutil` import from the test
module.

### WR-04: `protoc` compile had no timeout — a hostile/pathological schema could wedge the worker thread

**Files modified:** `src/kafka_mcp/adapters/outbound/schema_registry_http.py`
**Commit:** 0065786
**Applied fix:** Because WR-01 moved compilation in-process (a subprocess `timeout=` no longer
applies), the compile now runs on a `daemon` worker thread joined with a `_PROTOC_TIMEOUT_SECONDS`
(10s) budget in a new `_run_protoc_bounded` helper. On timeout a `RuntimeError` is raised, which the
existing `_decode_protobuf` handler wraps into a typed `DecodeError` (never an unhandled exception).
Compiler non-zero return codes and in-thread exceptions are likewise surfaced as `RuntimeError`. The
0700 `TemporaryDirectory`, fixed argv, and pinned `--proto_path` (no shell, no injection) are
preserved and documented inline.

### WR-05: `MessageToDict` default rendering diverged from the schema (camelCase), breaking nested Protobuf `value:<path>` matching

**Files modified:** `src/kafka_mcp/adapters/outbound/schema_registry_http.py`, `tests/test_adapters.py`
**Commit:** 0065786
**Applied fix:** Passed `preserving_proto_field_name=True` to `MessageToDict`, so the Protobuf decode
output uses the registered `.proto` snake_case field names — matching the Avro path so nested dotted
lookups (`value:payload.order_id`) resolve on Protobuf too. Added a test
(`test_decode_protobuf_preserves_snake_case_and_nested_paths`) with a nested message and an `int64`
field that asserts snake_case keys at both levels and verifies the `value:payload.order_id` and
`value:customer_id` dotted paths match end-to-end via `_matches_key`. (proto3 int64 still JSON-maps to
a string by spec; `_matches_key` compares via `str(...)`, so a string key still matches — documented
inline.)

### WR-02: `TransientError` was not exported from the package public API

**Files modified:** `src/kafka_mcp/__init__.py`
**Commit:** d124e09
**Applied fix:** Added `TransientError` to the `from kafka_mcp.domain.errors import (...)` block and to
`__all__`, alongside its sibling domain errors, so `from kafka_mcp import TransientError` works.

### WR-03: New `TransientError` inbound-face mapping had no test coverage

**Files modified:** `tests/test_inbound.py`
**Commit:** e35ffd9
**Applied fix:** Extended `MockKafkaClient.get_message` to raise `TransientError` for a `"transient"`
topic, imported `TransientError`, and added three face-level tests:
`test_fastapi_post_get_message_503_transient_error` (REST → HTTP 503 with `error == "TransientError"`),
`test_cli_get_message_transient_error_exits_3` (CLI → `SystemExit(3)`), and
`test_mcp_get_message_transient_error_mapped` (MCP → raises). These guard the boundary side of the
WR-05 fix against clause re-ordering or a 404 remap.
