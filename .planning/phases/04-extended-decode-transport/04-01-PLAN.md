<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Wire → domain model | Raw key bytes from the Kafka broker enter KafkaMessage.raw_key; bytes are attacker-controlled |
| model_dump() output | New bytes field raw_key flows into serialization; inbound adapters must base64-encode it (Plan 02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | KafkaMessage.raw_key (bytes field) | accept | raw_key is stored as opaque bytes; no interpretation in this plan. Framing validation and bounded length checks occur in Plan 02 service helpers (_extract_schema_id, _decode_key). Storing raw bytes in the model is safe — no execution or unbounded allocation. |
| T-04-02 | Denial of Service | KafkaMessage model with large raw_key | accept | raw_key mirrors the existing raw: bytes field; librdkafka enforces message.max.bytes at the broker/client level. No additional risk from adding the field. |
| T-04-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan. Existing dependencies unchanged. |

Note: framing guard (`len >= 5`, `raw[0] == 0x00`) is applied in Plan 02 service layer,
not here — keeping the adapter dumb (byte threading only) maintains hexagonal boundary.
</threat_model>
