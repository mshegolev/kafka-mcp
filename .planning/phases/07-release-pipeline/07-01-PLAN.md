Create `.github/workflows/release.yml` with the following specification:

**Triggers:**
- `push.tags: ["v*"]` — fires when a v* tag is pushed
- `workflow_dispatch` — manual trigger for dry-run testing

**Jobs:**

1. **`build`** job (runs always):
   - `runs-on: ubuntu-latest`
   - Checkout, setup Python 3.12
   - `pip install hatch twine`
   - `hatch build` — produces sdist + wheel in `dist/`
   - `python -m twine check dist/*` — validates distribution metadata
   - Upload `dist/` as artifact named `release-dist`

2. **`testpypi-upload`** job (runs always, needs build):
   - `runs-on: ubuntu-latest`
   - Download artifact `release-dist` into `dist/`
   - `python -m twine upload --repository testpypi dist/*`
     using env vars `TWINE_USERNAME: __token__` and
     `TWINE_PASSWORD: ${{ secrets.TEST_PYPI_TOKEN }}`
   - `--skip-existing` flag so re-runs don't fail on duplicate versions
   - `--verbose` for CI log diagnostics

3. **`pypi-publish`** job (runs ONLY on tag push, NOT on workflow_dispatch; needs testpypi-upload):
   - Condition: `if: github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v')`
   - `runs-on: ubuntu-latest`
   - `environment: pypi` (manual approval gate in GitHub repo settings)
   - `permissions: id-token: write` (OIDC Trusted Publisher)
   - Download artifact `release-dist` into `dist/`
   - Uses `pypa/gh-action-pypi-publish@release/v1` with
     `password: ${{ secrets.PYPI_TOKEN }}` and `skip-existing: true`

**Permissions:** Top-level `permissions: contents: read`. Only pypi-publish job
elevates to `id-token: write`.

**Security constraints:**
- Tokens ONLY from GitHub Secrets — never hardcoded
- Live PyPI publish gated on tag push + environment approval (human-gated)
- workflow_dispatch runs build + TestPyPI only (no live publish)
  </action>

**# Release Runbook — kafka-mcp**

Brief intro: this document covers how to publish a new kafka-mcp release.

**## Prerequisites**
- Python 3.10+ with `hatch` and `twine` installed (`pip install hatch twine`)
- Push access to the GitHub repository
- GitHub Secrets configured (see Secrets Setup below)

**## Tagging Convention**
- Tags follow semantic versioning: `v{MAJOR}.{MINOR}.{PATCH}` (e.g., `v1.1.0`)
- The tag MUST match the `version` field in `pyproject.toml`
- Update `pyproject.toml` version, `server.json` version, commit, then tag:
  ```bash
  # 1. Update version in pyproject.toml and server.json
  # 2. Commit the version bump
  git add pyproject.toml server.json
  git commit -m "chore: bump version to 1.1.0"
  # 3. Create and push the tag
  git tag v1.1.0
  git push origin main --tags
  ```

**## Secrets Setup**
Document the GitHub repository secrets required:
- `TEST_PYPI_TOKEN` — API token from https://test.pypi.org/manage/account/token/
  (scope: project `kafka-mcp` or entire account for first upload)
- `PYPI_TOKEN` — API token from https://pypi.org/manage/account/token/
  (scope: project `kafka-mcp`)
- How to add: GitHub repo → Settings → Secrets and variables → Actions → New repository secret
- Optional: configure `pypi` environment with manual approval in
  GitHub Settings → Environments → pypi

**## Local Verification**
Step-by-step local dry-run before tagging:
```bash
# Clean previous builds
rm -rf dist/

# Build sdist + wheel
hatch build

# Validate distribution metadata
python -m twine check dist/*
# Expected: PASSED for both .tar.gz and .whl

# (Optional) Test install from built wheel
pip install dist/kafka_mcp-*.whl --force-reinstall
python -c "from kafka_mcp import KafkaClient; print('OK')"
```

**## CI Pipeline (Automated)**
What happens when you push a tag:
1. `release.yml` triggers on the `v*` tag push
2. **Build job:** `hatch build` + `twine check dist/*`
3. **TestPyPI upload:** dry-run publish to https://test.pypi.org/project/kafka-mcp/
4. **PyPI publish:** (requires `pypi` environment approval) publishes to live PyPI
5. Verify on TestPyPI: `pip install -i https://test.pypi.org/simple/ kafka-mcp=={VERSION}`

Manual dry-run via workflow_dispatch (no tag needed):
- Go to Actions → "Release" → "Run workflow" → select branch → Run
- This runs build + TestPyPI upload only (no live PyPI publish)

**## Glama Submission**
After the PyPI package is live:
1. Ensure `glama.json` is up to date (tool list, transport config)
2. Go to https://glama.ai/mcp/servers and click "Submit Server"
3. Enter the GitHub repository URL: `https://github.com/mshegolev/kafka-mcp`
4. Glama reads `glama.json` from the repo root to populate server metadata
5. Verify the listing shows all tools (list_topics, describe_topic, search_messages,
   get_message, consumer_group_lag) and both transport options (stdio, streamable-http)

**## Troubleshooting**
Common issues:
- `twine check` fails: check `pyproject.toml` metadata (description, URLs, classifiers)
- TestPyPI upload 400 "file already exists": version already uploaded; bump version or use `--skip-existing`
- PyPI publish fails: verify `PYPI_TOKEN` secret is set and scoped to the project
- Glama submission rejected: verify `glama.json` schema matches https://glama.ai/mcp/schemas/server.json

**## Version History**

|---------|------|-------|
| v1.1.0 | TBD | consumer_group_lag tool, HTTP transport, real-broker E2E |
| v0.1.0 | 2026-06-08 | Initial v1.0 MVP release |

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI → PyPI | Build artifacts uploaded to external package registry |
| CI → TestPyPI | Dry-run upload to test registry |
| GitHub Secrets → CI | API tokens flow from repo settings to workflow |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Tampering | release.yml | mitigate | PyPI publish gated on tag push + `environment: pypi` manual approval; workflow_dispatch cannot trigger live publish |
| T-07-02 | Info Disclosure | secrets | mitigate | Tokens ONLY from `${{ secrets.* }}`; never hardcoded in workflow; `--verbose` does not leak token values (twine masks them) |
| T-07-03 | Elevation | publish job | mitigate | `id-token: write` granted only to pypi-publish job, not workflow-level; `permissions: contents: read` at workflow level |
| T-07-04 | Spoofing | tag trigger | accept | Any user with push access can create tags; mitigated by `environment: pypi` approval gate if configured |
| T-07-05 | Denial of Service | TestPyPI upload | accept | `--skip-existing` prevents failure on duplicate versions; TestPyPI is a dry-run target |
</threat_model>
